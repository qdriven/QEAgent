# v3.9.20 Release Notes

**Release Date:** 2026-05-08

## Highlights

A second wave of self-learning wire-gap fixes built on top of v3.9.19. HNSW A (the long-term pattern catalog) is now consulted on every `task_orchestrate` call, hook-side experience writes get embeddings inline instead of waiting for the next-boot backfill, dream cycles inherit the same patient SQLite timeout the rest of the system uses, pattern `quality_score` actually moves when patterns are reused through the hook flow, and the `TrajectoryBridge` no longer maintains its own `.agentic-qe/trajectories.db` outside the unified-memory rule.

## Fixed

- **Background ReasoningBank singleton was lazy-initialized at request time.** The `ReasoningBankService` singleton + `EnhancedReasoningBankAdapter.initialize()` + HNSW A index population happened on the first `task_orchestrate` call. The first request after a cold MCP start paid the full HNSW build cost on the request path, and racing calls fell through to context-only matches before the index was ready. Fix: fire-and-forget pre-warm in `src/mcp/entry.ts` after `CrossPhaseHooks` eager-init, so the warm-up runs in parallel with the rest of MCP startup.

- **`aqe_health` reported `vectorCount: 0` even when the vectors table had hundreds of rows.** `core-handlers.ts:597` probed `getVectorCount` on the unified memory manager, but the method is `vectorCount()` (`unified-memory.ts:755`). The mismatch made the `typeof` check fail silently. Fix: corrected the method name at both call sites.

- **HNSW A (`qe_patterns`) was never consulted on the routing decision path.** `handleTaskOrchestrate` already called `getExperienceGuidance()` to surface relevant past trajectories (HNSW C). The catalog of consolidated long-term patterns was only checked through unrelated MCP tools — never on the request that actually picked an executor. Fix: `searchPatterns(task, {limit: 5, domain})` runs in parallel with `getExperienceGuidance()`, and the top hits are forwarded as `patternHints` on the `submitTask` payload (`patternId`, `name`, `description`, `confidence`, `similarity`, `canReuse: tier === 'long-term'`).

- **Hook-side `captured_experiences` writers wrote rows with no embedding.** Both `persistCommandExperience` (sources: `cli-hook-post-command`, `cli-hook-post-edit`) and `persistTaskOutcome` (source: `cli-hook-post-task`) ran the SQL insert and returned without computing an embedding. ~85% of new captured-experience writes since v3.9.19 came through these two hook paths and contributed zero embeddings, leaving HNSW C cold for hook-driven workloads. Fix: both writers now fire a no-await IIFE after the insert that imports `computeRealEmbedding`, computes `text = ${domain}: ${task}`.slice(0, 512), and `UPDATE`s the row in place. Fail-soft — missed embeddings are recoverable via the boot backfill below.

- **`ExperienceReplay.initialize()` left ghost rows in the index.** The HNSW C load loop only added rows that already had embeddings; nothing backfilled the rows that hook paths left empty (~63% of the table in one observed pulse). Fix: a cap-200 fire-and-forget IIFE in `initialize()` selects ghost rows, computes embeddings via the same real-embeddings config the canonical writer uses, `UPDATE`s in place, and adds the new vectors to the live HNSW index so they're searchable without restart.

- **`qe_trajectories` had no embeddings at all.** `TrajectoryTracker.endTrajectory()` writes the row with `embedding = NULL` and there's no follow-up worker. Hook-side trajectory inserts left the column NULL too. Observed: 719 hook-created rows, 0 embeddings — kNN over historical trajectories collapsed to "no candidates". Fix: a second cap-200 fire-and-forget IIFE in `ExperienceReplay.initialize()` adjacent to the captured-experiences backfill, scoped to `qe_trajectories`. PRAGMA-checks the embedding column first since `TrajectoryTracker.ensureSchema()` is what adds it; no-ops if it hasn't run yet.

- **`DreamEngine` opened a fresh DB handle that never inherited the busy_timeout pragma.** `checkAndTriggerDream()` dynamically imports `createDreamEngine`, which goes through `getUnifiedPersistence().getDatabase()`. The hook-side and worker-side busy_timeout pragmas (patches 260/270) never reached that dynamically-imported path — it ran with the platform default 0ms timeout. Under concurrent MCP-worker contention on the WAL write-lock, `SQLITE_BUSY` hit immediately and cycles were marked failed (199/279, 71%, in one session). Fix: applies `pragma('busy_timeout = 60000')` in `DreamEngine.initialize()` so dream cycles wait patiently like the rest of ADR-001 Option C.

- **`recordUsage()` was unreachable from the hook flow.** `SQLitePatternStore.recordUsage()` updates `qe_patterns.{usage_count, successful_uses, success_rate, quality_score}` but is only invoked through the `HandleTaskOutcomeRecord` MCP path. The hook flow that fans out per-pattern `experience_applications` rows (patches 160 + 300) never called it — `quality_score` stayed pinned at the bootstrap value (~0.30 floor in observed data, with 88/89 patterns never crossing into long-term). Fix: inline SQL inside the `persistTaskOutcome` bridge loop after the per-pattern `experience_applications` insert, using the same formula as `recordUsage()`: `confidence * 0.3 + min(usage_count/100, 1) * 0.2 + success_rate * 0.5`. Reads `confidence` from the pattern row first because `UPDATE` expressions can't reference it directly.

- **`TrajectoryBridge` kept its own `.agentic-qe/trajectories.db` outside unified memory.** `src/adapters/claude-flow/trajectory-bridge.ts:184` opened a separate SQLite file with its own `trajectories` schema, violating the project's "all data goes through SQLite (better-sqlite3) — one DB, one schema" rule. Feedback written via `TrajectoryBridge.endTrajectory()` never showed up in any of the queries that read `qe_trajectories`. Fix: `persistTrajectory()` now opens the unified memory DB via `getUnifiedMemory()`, maps the bridge's columns onto `qe_trajectories` (started/completed epoch ms → ISO datetime), and writes to the same row everyone else reads. Adds the `feedback` column defensively in case `TrajectoryTracker.ensureSchema()` hasn't run yet on a fresh install. The legacy `.agentic-qe/trajectories.db` path stays in `init.ts` `reset --all` so upgraded installs still clean up the old file.

- **Claude provider implementations ignored `ANTHROPIC_BASE_URL`.** `src/coordination/consensus/providers/claude-provider.ts:181` and `src/shared/llm/providers/claude.ts:383` hardcoded the `api.anthropic.com` fallback when no `baseUrl` was passed in config, so AQE LLM calls bypassed proxies (Claude Max Proxy, LiteLLM, etc.) that the rest of the codebase honors via `process.env.ANTHROPIC_BASE_URL`. Fix: both providers now consult the env var before falling through to the literal hostname, mirroring the pattern already used by `context/compaction/llm-caller-adapter.ts:23`. No-op when the env var is unset.

## Added

- **Opt-in trajectory judge.** New `src/mcp/handlers/trajectory-judge.ts` runs at the end of `handleTaskOrchestrate` (fire-and-forget) when `AQE_TRAJECTORY_JUDGE=1` and `ANTHROPIC_API_KEY` are both set. Picks the 5 most-recent `qe_trajectories` rows where `feedback IS NULL AND ended_at IS NOT NULL`, asks Claude Haiku for a structured `{quality, reasoning, improvement?}` verdict, and writes the JSON into `qe_trajectories.feedback`. Quality is embedded in the JSON to avoid a schema migration. Bounded to 5 rows per call so cost is predictable. Uses the proxy-aware base URL from the same release.

## Upgrade Notes

- No breaking changes for users on `npx agentic-qe`.
- Pre-existing installs with `.agentic-qe/trajectories.db` will see that file go stale — the live writer now targets `memory.db`. `aqe init reset --all` still cleans up the legacy file.
- `AQE_TRAJECTORY_JUDGE=1` is opt-in by design because each `task_orchestrate` call triggers up to one Claude Haiku request when there are unjudged trajectories. Cost is ~5× a 200-token Haiku call per `task_orchestrate` in the worst case.
- The HNSW A pattern hints introduce a single additional `searchPatterns` call per `task_orchestrate`. Fail-soft on errors so a misbehaving HNSW A path can't block task execution.
